## **文本摘要**

与翻译任务类似，摘要也属于 **序列到序列（Seq2Seq）** 任务，但其特点在于输入通常很长（文章或评论），而输出（摘要）则相对短促且高度浓缩。
本节以`mT5` 模型和多语言亚马逊评论数据集（MARC）为例进行了讲解。

---

### 1. 总结

本节介绍了如何微调一个**生成式摘要模型**。重点讨论了如何处理长文本输入、如何使用特定的分词技巧（如为 T5 类模型添加前缀），
以及如何通过特定的评估指标（如 ROUGE）来衡量摘要的质量，而不仅仅是看训练损失。

---

### 2. 处理数据集的流程

摘要任务的预处理流程需要平衡“信息量”和“计算开销”：

1. **数据清洗与筛选：** 摘要任务对数据质量很敏感。文中演示了如何过滤掉长度过短的摘要，以确保模型学习的是有意义的压缩。
2. **分词处理：**
    * **输入（Inputs）：** 对原始长文本进行分词，通常设置较大的 `max_length`（如 512 或 1024）。
    * **目标（Targets）：** 使用 `text_target` 参数对摘要文本进行分词，设置较小的 `max_length`（如 128）。
3. **特殊前缀（针对 T5）：** 如果使用 T5 系列模型，通常需要在输入文本前添加 `summarize: ` 这样的前缀，告诉模型当前执行的任务类型。
4. **动态填充与准备（Data Collator）：**
    * 使用 **`DataCollatorForSeq2Seq`**。
    * 它会自动处理 `input_ids` 和 `labels`。
    * 最重要的是，它会为解码器准备好 **`decoder_input_ids`**。

---

### 3. 如何计算损失值 (Loss)

摘要模型的训练目标是让模型生成的 Token 序列与参考摘要尽可能一致。

* **计算机制：** 采用 **交叉熵损失（Cross-Entropy Loss）**。
* **计算逻辑：** 在解码器的每一个步骤，模型会输出一个覆盖整个词表的概率分布。计算 Loss 时，会将这个分布与该位置对应的“参考答案”索引进行对比。
* **平滑与掩码：** 填充位（Padding）会被屏蔽掉，不贡献任何 Loss。

---

### 4. 选取什么数据作为训练时的参考 Labels

在摘要任务中，参考标签（Labels）直接来源于数据集中的**目标摘要文本**。

* **数据选取：** 经过分词器处理后的摘要 Token ID 序列。
* **处理细节：**
    * 标签序列中必须包含“序列结束符”（`</s>` 或 `EOS`），这教会模型在完成摘要后主动闭嘴。
    * 如果摘要太长，会被截断；如果太短，在 Batch 计算时会被填充。

---

### 5. 训练过程中 “参考答案” 是什么

在摘要训练的黑盒子里，“参考答案”是经过精心加工的 **`labels` 张量**：

1. **形态：** 一个与解码器输出长度对齐的整数张量。
2. **特殊占位符 `-100`：**
   为了让模型只关注真正的摘要内容，**所有填充（Padding）位置的 ID 都会被替换为 `-100`**。
    * *原理：* PyTorch 的 Loss 函数会自动忽略索引为 `-100` 的位置。这意味着模型预测“填充符”预测得准不准，完全不影响它的成绩。
3. **Teacher Forcing 机制下的参考：**
   在训练时，虽然我们的参考答案（Labels）是完整的摘要，但解码器实际上是在玩“接龙”。
    * 给它：`[起始符] + 摘要的前 n 个词`。
    * 让它预测：`第 n+1 个词`。
    * **参考答案**：就是那个真实的 `第 n+1 个词`。

---

### 💡 提示：摘要与翻译的区别

虽然流程相似，但摘要任务在评估时更依赖 **ROUGE 指标**（关注召回率，即参考摘要中有多少词出现在了模型生成的摘要中），而翻译更看重
**BLEU 指标**（关注精确率）。这是因为摘要的表达方式非常灵活，只要抓住了关键点，即便措辞不同也是好摘要。
